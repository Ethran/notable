  extra clicks) while being efficient. This is a common pattern in modern apps.

  Current Usage                                                                 
                                                                                
  The codebase currently uses two versions of getFile():                        
                                                                                
  1. getFile(path): ByteArray - Downloads entire file into memory               
    - Used for: JSON files (manifests, deletions.json, page JSONs)              
    - These are all small files (< few MB)                                      
  2. getFile(path, localFile: File) - Downloads and saves directly to disk      
    - Used for: Images and background files (line 680, 703)                     
    - These can be large, so they're saved directly to disk                     
                                                                                
  Why getFileStream() Exists (But Isn't Used)                                   
                                                                                
  getFileStream() was likely added for future scenarios where you might want to:
  - Stream a file for processing without loading it all into memory             
  - Show download progress                                                      
  - Process data incrementally (e.g., parsing a huge file chunk by chunk)       
                                                                                
  Why We Fixed It Anyway                                                        
                                                                                
  Even though it's not used, the maintainer spotted the bug during code review. 
  This is important because:                                                    
                                                                                
  1. It's a latent bug - If someone tries to use it in the future, it would leak
   resources                                                                    
  2. Good API hygiene - Public methods should be correct even if unused         
  3. Documentation value - The code review comment flags it for anyone reading  
  the code                                                                      
                                                                                
  The Actual Bug                                                                
                                                                                
  Before:                                                                       
  fun getFileStream(path: String): InputStream {                                
      val response = client.newCall(request).execute()                          
      return response.body?.byteStream() ?: throw IOException()                 
      // BUG: response object stays open forever! ☠️                            
  }                                                                             
                                                                                
  The caller gets an InputStream but has no way to close the underlying HTTP    
  Response. This causes:                                                        
  - Connection pool exhaustion                                                  
  - Memory leaks                                                                
  - Eventually: "Too many open connections" errors                              
                                                                                
  After:                                                                        
  fun getFileStream(path: String): StreamResponse {                             
      val response = client.newCall(request).execute()                          
      return StreamResponse(response, inputStream)  // ✓ Wrapper manages both   
  }                                                                             
                                                                                
  Now the caller can do:                                                        
  webdavClient.getFileStream(path).use { streamResponse ->                      
      // Work with streamResponse.inputStream                                   
  }  // Automatically closes BOTH stream AND response                           
                                                                                
  Bottom Line                                                                   
                                                                                
  You don't need to test StreamResponse because nothing uses it. We fixed it    
  for:                                                                          
  - Code correctness (preventing future bugs)                                   
  - Addressing maintainer feedback (showing we're thorough)                     
  - Good engineering practice (fixing broken code even if unused)               
                                                                                
  It's like fixing a broken fire extinguisher - you hope you never need it, but 
  if you do, you want it to work!

  reMarkable's Approach                                                         
                                                                                
  What reMarkable does well:                                                    
  - Made text conversion optional (not forced)                                  
  - Works on selection (lasso tool selects strokes)                             
  - Converts to editable text (can fix errors)                                  
  - Keeps original handwriting alongside conversion                             
                                                                                
  You could replicate this:                                                     
  // User taps "Extract Text" button                                            
  fun extractTextFromSelectedStrokes(selectedStrokes: List<Stroke>) {           
      val text = recognizeText(selectedStrokes)  // MyScript or ML Kit          
                                                                                
      // Show preview dialog                                                    
      showTextPreview(                                                          
          originalHandwriting = renderToBitmap(selectedStrokes),                
          recognizedText = text,                                                
          onAccept = { editedText ->                                            
              insertIntoMarkdown(editedText)                                    
          }                                                                     
      )                                                                         
  }
